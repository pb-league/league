01

Architecture Overview

Pickleball League Manager uses a two-tier architecture with no dedicated server infrastructure.

ComponentDescription
FrontendStatic HTML/CSS/JS hosted on GitHub Pages. Three pages: index.html (login), player.html (player dashboard), admin.html (admin dashboard).
BackendGoogle Apps Script Web App (Code.gs) serving as a REST-like API. Deployed as a public web app executing as the script owner.
DataGoogle Sheets. One master registry sheet lists all leagues. Each league has its own sheet with tabs: config, players, attendance, pairings, scores.
settings.jsContains GAS_URL, APP_VERSION, APP_BUILD_DATE. Must be updated with your deployed GAS URL before going live.
Performance
GAS has a cold-start delay of 2–4 seconds when idle. A scheduled warmup() trigger every 5 minutes keeps the script warm. URL-based logins (?league=slug) use a single combined endpoint to cut round-trips in half.
02

Login & Authentication

Player Login

Three-step wizard: select league → select name → enter PIN. URL parameters can skip any or all steps and can be combined freely:

ParameterEffectExample
?id=Filters the league dropdown to leagues belonging to that organisation (plus unassigned leagues)?id=springfield
?league=Pre-selects a specific league by its ID slug or display name, skipping step 1?league=tuesday-doubles
?player=Pre-selects a player by name, skipping step 2 and jumping straight to PIN entry?player=Doug

All three can be combined in one URL. For example:

index.html?id=springfield&league=tuesday-doubles&player=Doug

This filters to Springfield leagues, selects Tuesday Doubles, pre-fills Doug's name, and drops straight to the PIN screen — one tap from a bookmark or home-screen shortcut.

Find My League

If a player arrives without a ?id= URL and doesn't see their league in the dropdown, a Find My League section appears below the dropdown. They can enter either their Organisation ID or a partial league name (or both) and click the search button. Matching leagues are added to the dropdown and auto-selected if only one is found.

Admin Login

Clicking 🔐 Admin Login opens a dedicated password screen — completely separate from the player PIN system. Two tabs:

PathPassword locationAccess
League AdminConfig sheet — adminPin fieldFull league control
App ManagerGAS Script Properties — APP_MANAGER_PASSWORDAll leagues, limits, customer IDs
Security
No player can accidentally gain admin access by choosing the same PIN as the admin password — the two systems are completely separate.

Admin Assistants

Assistants log in through the regular player PIN path (not Admin Login). Because their role is set to assistant in the Players page, the app automatically routes them to admin.html with restricted access.

04

Deployment & Setup

  1. Paste Code.gs into a new Google Apps Script project.
  2. Update REGISTRY_SHEET_ID in Code.gs to your master registry Google Sheet ID.
  3. Add columns 7–13 to the registry sheet: createdDate, expiryDays, maxPlayers, maxCourts, maxRounds, maxSessions, customerId.
  4. Deploy as Web App: Execute as Me, Anyone can access.
  5. Copy the Web App URL into js/settings.js as GAS_URL.
  6. In Script Properties, add APP_MANAGER_PASSWORD = your chosen App Manager password.
  7. Set up a time-based trigger: function warmup(), every 5 minutes.
  8. Push all files to GitHub Pages (or any static hosting).
  9. Open index.html and use admin.html?setup=1 to create your first league.
Important
Set the Google Sheets to private — only the script owner needs access. Players never interact with the sheet directly.

07

Self-Registration

Enable on the Setup page with Allow self-registration. A Register as New Player button appears on the login page.

Registration Flow

  1. Player clicks Register, enters the Invite Code (provided by admin), their handle, email (optional), gender group, notification preference, PIN, and role (Player or Spectator).
  2. GAS validates the code, checks the pending cap, and checks for duplicate handles.
  3. New player is written to the sheet as inactive.
  4. Admin receives a notification email (if Reply-To is configured).
  5. New player appears in the Players list with status Pending (PEND) — they cannot log in yet.
  6. Admin changes the player's status from Pending to Active on the Players page.
  7. If the player provided an email, an approval email is sent automatically with a Log In Now button linking to the league URL (set in Setup).
Pending dashboard tile
A gold Pending Approval stat tile appears on the admin Dashboard whenever there are players awaiting approval, so registrations are never missed.

Anti-Abuse Protections

ProtectionHow it works
Invite CodeRequired to register. Set in Setup. Case-insensitive. Rotate if abused.
Max Pending CapRejects new registrations if pending count reaches the limit (default 10).
Admin NotificationEmail sent to Reply-To address for each registration, with pending count.
Duplicate CheckCase-insensitive handle uniqueness enforced server-side.
08

Multi-League & Customer IDs

The app supports unlimited leagues from one deployment. The master registry Google Sheet lists all leagues.

Customer ID Filtering

Each league in the registry can have a Customer ID (col 13). This controls which leagues appear in the login dropdown:

  • index.html?id=abc — shows leagues with customerId = abc plus leagues with no ID
  • No ?id= — shows only leagues with no customer ID assigned

This lets you give each customer a URL that shows only their leagues, while shared/public leagues remain visible to all.

New Customer Setup

  1. App Manager assigns a customer ID (e.g. their organisation slug).
  2. Creates an initial league with that ID via the All Leagues page.
  3. Shares the URL: index.html?id=<customerId>
  4. All leagues the customer creates inherit their ID automatically.
Note
Customer IDs are a display filter, not a security mechanism. Anyone who knows a league's slug can access it via ?league=slug.

Hidden Leagues

Set Hidden in the All Leagues table to exclude a league from all dropdowns. It's still accessible via ?league=slug in the URL. Useful for test leagues or private groups.

Attendance & Sit-Out

Mark attendance before generating pairings. Only In players are included in generation.

StateMeaning
InPresent and included in pairing generation.
OutAbsent for this session.
TBDStatus not yet set — not included in generation.
Sit OutPresent but temporarily sitting out mid-session. Stays in standings and history. Excluded from pairing regeneration until switched back to In.

Click any attendance cell to cycle through states. Changes save automatically. Spectator-role players only cycle between Out and TBD. The Players In totals row at the bottom of the grid updates live as you click cells — no page refresh needed.

Mid-session sit-out
If a player needs to sit out mid-session, mark them Sit Out and regenerate the remaining rounds. They'll be excluded from the new pairings but their existing scores and standings are preserved.
11

Generating Pairings

The Pairings page has three collapsible sections: Generate Pairings, Tournament Mode, and Edit / Add Pairing.

Generate Pairings Flow

  1. Mark attendance for the session.
  2. Select scope: All rounds, All remaining rounds, or a specific round number.
  3. Click 🎲 Generate Best Pairings. A spinner shows while the optimizer runs.
  4. Review the proposed pairings in the preview below.
  5. Click 🔒 Lock & Save to commit them to the sheet.
Warning
Pairings are not saved until you click Lock & Save. Navigating away will lose unsaved pairings.

Partial Round Generation

You can generate individual rounds independently. The optimizer pre-warms from already-locked rounds, so it's aware of partners and opponents from earlier rounds when generating later ones.

Editing Pairings

Use Edit / Add Pairing to manually create or modify individual games. Select session, round, and court, then click Load to edit an existing pairing or fill in new values to add one.

Clearing Pairings

Use 🗑 Clear Rounds to remove generated pairings. Scope selector controls which rounds are cleared. Existing scores for cleared rounds are also removed.

12

Pairing Algorithm

The optimizer evaluates up to 500 random candidate solutions and keeps the best one. Each candidate is scored by a weighted sum of penalty factors — lower score = better pairing.

FactorDefault WeightDescription
Session Partners50Penalty for pairing players who've already partnered this session.
Session Bye30Penalty if a player already had a bye this session gets another.
Bye Variance20Penalises uneven bye distribution within the session.
Session Opponents20Penalty for repeat opponents within the session.
Rank Balance15Penalises mismatched team or opponent skill levels.
History Partners10Penalty based on full-season repeat partner history.
Rank Std Dev8Penalises games where all four player ranks are very spread out.
History Opponents3Penalty based on full-season repeat opponent history.

All weights are configurable in Setup → Optimizer Weights. Set to 0 to ignore a factor entirely.

Weight Calibration

Raw penalty values are on different scales. Before scoring, each weight is automatically calibrated by sampling real distributions so that all factors contribute proportionally to their configured importance.

Ranking Integration

Player ranks feed into pairing decisions. For players without season standings, Init Rank provides a fallback. Players with no rank get the median rank position.

Bye Assignment

Players with the fewest session byes sit out first. Season bye count is the secondary tiebreaker. Any players left over after courts are filled also receive byes automatically.

Game Formats

Set in Setup → Game Mode. The optimizer adapts its pairing logic for each format.

FormatDescription
DoublesFour-player games. Full partner and opponent history tracked across the season.
Mixed DoublesSame as Doubles with an additional gender-balance penalty encouraging mixed-gender partnerships. Set player Group (M/F/Either) on the Players page.
SinglesOne vs one. Opponent history tracked to distribute matchups evenly. Rank-balance weighting keeps games competitive.
Fixed-Pair DoublesPermanent partnerships entered as a single player record (both names together). Scheduled as a singles-style unit — pairs compete against other pairs. No re-pairing occurs.
14

Score Entry

Scores are entered on the Score Entry page. Each round is a collapsible section showing a badge with how many scores are entered vs remaining.

  • Each round is collapsible — a ▲/▼ triangle on the left of the round header toggles it. Rounds auto-collapse when all scores are entered, with a ✓ badge showing the count.
  • The Generate Pairings button always shows the current session number (e.g. Generate Pairings for Session 3) so it's clear which session will be affected.
  • A 🔄 Refresh button fetches the latest scores from the sheet in case players have entered scores since the page loaded.
  • When switching sessions, a loading indicator appears immediately so you know the change is registered while the data fetches.
  • If existing scores would be overwritten, a warning dialog shows the difference before saving.
  • Tied scores trigger a confirmation before saving.
Players entering scores
Players with Scorer role or Can Score enabled can enter scores from the player dashboard. Their scores are merged with existing scores — they cannot clear or overwrite other entries.
15

Tournament Overview

Tournaments run within a regular session. Tournament pairings are saved as type tourn-game, which distinguishes them from regular round-robin pairings in standings and displays.

All three modes use the same seeding priority: season standings rank → Init Rank → alphabetical.

General Flow

  1. Mark attendance for the session.
  2. Open Tournament Mode on the Pairings page.
  3. Select format and click 🏆 Generate Bracket.
  4. Click 🔒 Lock & Save Round to save the round.
  5. Enter scores, then click ▶ Advance Round to generate the next round.
  6. Repeat until the tournament is complete.

Tournament results are available in the Tournament item under Reports in the nav.

15a

Single Elimination

One loss eliminates a player or team. Fastest format — ideal for end-of-season finales.

  • Seed 1 vs lowest seed, seed 2 vs second-lowest, etc.
  • Top seed receives a bye if player count is odd.
  • For doubles: players are paired into fixed teams before the bracket — seeds 1+2 form Team 1, seeds 3+4 form Team 2, etc. Teams stay fixed throughout.
  • Tournament ends when one player or team remains.
15b

Double Elimination

Two separate brackets — Winners and Losers. Players need two losses to be eliminated.

  • First loss: player moves from Winners to Losers bracket.
  • Second loss in Losers bracket: eliminated.
  • Grand Final: when exactly one player remains in each bracket, they meet. The Losers bracket champion must win to force a potential second game.
  • Status panel shows live counts: Winners bracket, Losers bracket, Eliminated.
Doubles
Same as Single Elimination — teams are fixed at the start and remain together throughout both brackets.
15c

Round Robin (Reseeded)

Everyone plays every round — no eliminations. The player with the most wins at the end is champion. Ideal for odd player counts since the bye rotates automatically.

  • Bye rotation: the player with the fewest byes so far sits out. Among ties, the lowest-ranked player (weakest) gets the bye.
  • Matchups: after sorting by current wins (initial seed as tiebreak), rank 1 plays rank last, rank 2 plays rank second-last, etc.
  • Doubles re-pairing: teams reform each round from current rankings — rank 1+2 form one team, rank 3+4 another, etc. This is necessary so the bye recipient can be reintegrated the next round.
  • No automatic end: the admin runs as many rounds as desired and stops when done.

The status panel shows a live standings table (rank, wins, losses, byes) updated after each round.

16

Standings

Three tabs on the Standings page:

  • Season Standings — all games across the full season. Primary sort: win percentage. Secondary sort: configurable (avg point differential or points percentage).
  • Session Standings — wins and losses for a single selected session.
  • Ranking Trend — chart showing each player's rank changing across sessions.

Players with zero games played and inactive players are excluded from standings. Sub-role players are also excluded.

Avg+/- explained
Hovering over the Avg+/- column header shows a tooltip: it is the average score differential per game (your team's average score minus the opponent's average score). A positive value means you outscore opponents on average. It is used as a tiebreaker when two players have identical win percentages.

Player Reports

Found under Reports → Player Report. When a player opens this page, it automatically defaults to their own name and loads their report immediately — no manual selection needed. They can then choose any other player from the dropdown.

Shows for a selected player:

  • Current rank and win/loss record
  • Faced as Opponent table — every opponent with game count and W/L record against them
  • Played as Partner table — every partner with game count and W/L record together
  • Full game log with session, round, partner, opponents, score, and result

Click 📧 Email Report to send the report directly to the player's email address. Spectators and Subs are excluded from the player report dropdown.

Head-to-Head Analysis

Found under Reports → Head-to-Head. Shows how often every pair of players has faced each other, across two tabs.

Partners Tab

A heat-map matrix showing how many times each pair of players has been partners. Darker green = more games together. Below the matrix, an Average Partner Rank table ranks every player by the average rank of their partners (weighted by games played). A player near the top has consistently been paired with higher-ranked partners.

Opponents Tab

Same matrix for opponents. The Average Opponent Rank table shows who has faced the toughest opposition on average. In doubles, each game has two opponents — the table correctly counts each game once (not twice).

Average Rank Calculation

For each player: multiply each partner's/opponent's rank by the number of games played with them, sum those values, then divide by total games. Ranks come from current season standings, with Init Rank as fallback, and median rank for anyone without a rank yet.

Both tabs have a session filter — select a specific session or all sessions to see how pairings evolved over the season.
